#### 步骤 4: 选择设计概念

### 设计方案评估与选择

#### 方案一：单体架构（Monolith）

- 所有功能打包在一个Java应用中
- 使用Spring Boot内建MVC、JPA、REST
- 数据库直连，定时任务处理价格更新
- 前端Angular打包进静态资源

✅ 优点：

- 开发简单，适合MVP快速上线（CON-4）
- 易于调试和部署初期版本
- 符合团队熟悉技术栈

❌ 缺点：

- 难以满足Q-1（<100ms发布）和Q-4（可扩展性）
- 价格变更与查询耦合，影响SLA
- 不利于未来gRPC扩展（Q-6）
- 可靠性风险集中（Q-2）

#### 方案二：分层+事件驱动微服务架构

- 分离为多个轻量服务：API Gateway、Price Service、Config Service、Auth Adapter、Pricing Engine（独立进程）、Event Publisher
- 使用Kafka实现价格变更事件广播
- 查询走缓存（Redis），写入触发异步计算与发布
- 前后端分离，Angular独立部署

✅ 优点：

- 天然支持Q-1（异步计算+流水线优化）
- 支持Q-2（Kafka持久化确保100%送达）
- 满足Q-3/Q-4（读写分离，缓存提升可用性与扩展性）
- 支持Q-6（新协议可通过Sidecar或独立服务接入）
- 利于CRN-2（重用Kafka技能）、CRN-5（容器化CI/CD）

❌ 缺点：

- 初期复杂度略高
- 需要协调多个服务间通信

#### 方案三：Serverless 架构（基于FaaS）

- 使用Lambda函数处理价格变更、查询请求
- 状态存储于DynamoDB/Redis
- 事件由API Gateway和Kafka触发

✅ 优点：

- 极致弹性，自动扩缩容（Q-4）
- 按需付费，运维成本低

❌ 缺点：

- 冷启动延迟难以满足<100ms（Q-1）
- 调试困难，不利于Q-9（可测试性）
- 团队无相关经验（违背CRN-2）
- 不利于长期维护与知识传承

---

### 最终选择：**方案二 —— 分层+事件驱动微服务架构**

**优势总结**：

- 在性能、可靠性、可用性之间取得最佳平衡
- 充分利用团队已有Java/Spring/Kafka/Angular技能（CRN-2）
- 支持云原生部署（CON-6）
- 模块清晰，便于CRN-3（团队分工）和CRN-4（避免技术债务）
- 为后续迭代预留演进路径（如引入gRPC、AI预测模块）

#### 步骤 5: 实例化架构要素

### 实例化架构要素：组件、职责、接口定义

#### 一、容器级组件划分

| 容器名称                        | 技术栈                                        | 职责                                                           |
| ------------------------------- | --------------------------------------------- | -------------------------------------------------------------- |
| **HPS Web UI**                  | Angular 16+, TypeScript, RxJS                 | 提供用户界面：登录、价格变更、查询、管理操作；调用后端REST API |
| **API Gateway**                 | Spring Cloud Gateway                          | 请求路由、认证拦截、限流熔断、协议转换                         |
| **Auth Service Adapter**        | Java/Spring Boot, OAuth2 Client               | 对接云身份服务（OIDC），获取用户信息与权限                     |
| **Hotel & Rate Config Service** | Java/Spring Boot, JPA/Hibernate               | 管理酒店、房型、税率、基础房价规则（HPS-4/5/6）                |
| **Price Calculation Engine**    | Java/Spring Boot, Rule Engine (Drools可选)    | 接收基准价变更事件，执行衍生房价计算逻辑（动态包价、折扣等）   |
| **Price Query Service**         | Java/Spring Boot, Spring Data Redis           | 提供高性能价格查询API，从缓存读取最新价格                      |
| **Event Publisher**             | Java/Spring Boot, Kafka Producer              | 将价格变更事件发布至Kafka Topic（price-updated）               |
| **Message Broker**              | Apache Kafka                                  | 消息队列，持久化价格变更事件，保证可靠投递                     |
| **Channel Listener**            | Java/Kafka Consumer                           | 订阅price-updated事件，推送至渠道管理系统（CMS）               |
| **Data Stores**                 | PostgreSQL（配置数据）、Redis（实时价格缓存） | 持久化结构化配置；高速缓存当前价格                             |

#### 二、关键接口定义（REST 示例）

```http
POST /api/v1/prices
Content-Type: application/json
{
  "hotelId": "HOTEL_001",
  "baseRate": 299.0,
  "effectiveDate": "2025-04-05"
}
```

→ 触发同步校验 → 发布事件 → 异步计算衍生价 → 更新缓存 → Kafka广播

```http
GET /api/v1/prices?hotelId=HOTEL_001&date=2025-04-05
→ 返回JSON格式价格快照（含基准价与所有衍生价）
```

#### 三、事件契约（Kafka）

**Topic**: `price-updated`  
**Key**: hotelId + date  
**Value (JSON)**:

```json
{
  "eventId": "evt-123",
  "hotelId": "HOTEL_001",
  "baseRate": 299.0,
  "effectiveDate": "2025-04-05",
  "timestamp": "2025-04-01T10:00:00Z",
  "version": 1
}
```

### 评估的多个设计方案，以及最终选择的方案及其优势

#### 方案评估：价格变更处理架构

| 方案                                                          | 描述                                                                | 优点                             | 缺点                                   | 适用质量属性                 |
| ------------------------------------------------------------- | ------------------------------------------------------------------- | -------------------------------- | -------------------------------------- | ---------------------------- |
| **A. 同步处理链**<br>(HTTP调用串联计算+发布)                  | 更改价格 → 计算衍生价 → 同步发布到渠道                              | 简单直观，调试容易               | 单点故障；超时风险；无法满足<100ms要求 | ❌ 不满足Q-1/Q-2             |

| **D. Serverless 函数编排**<br>(AWS Lambda / Azure Functions)  | 每个步骤封装为函数，由事件触发                                      | 弹性伸缩、按需付费               | 冷启动延迟、跨函数状态管理难           | ⚠️ 与团队技术栈不符（CRN-2） |

#### 最终选择：**方案 B —— 异步事件驱动 + Kafka**

> ✅ **优势总结**：

- 利用 Kafka 提供：
  - **可靠性**：持久化事件，支持重放（Q-2）
  - **性能**：异步解耦，主流程响应快（Q-1）
  - **可扩展性**：消费者可水平扩展应对高峰（Q-4）
- 符合团队已有 Kafka 技术储备（CRN-2）
- 成熟模式，易于维护与监控

> 🔄 补充策略：结合轻量级 CQRS 模式，实现读写分离，进一步优化查询路径。

#### 步骤 4: 选择设计概念

### 设计方案评估与选择

#### 方案一：同步HTTP调用 + 数据库轮询补偿

- **描述**：价格变更后同步调用各渠道API，失败则写入DB由定时任务重试
- **优点**：实现简单，易于调试
- **缺点**：无法满足<100ms性能要求；轮询延迟高；难以保证一致性
- **适用性**：❌ 不满足Q-1/Q-2

#### 方案二：异步消息队列 + 幂等消费者 + 死信队列

- **描述**：使用Kafka作为主事件总线，所有变更发布为事件；下游消费端幂等处理；失败转入DLQ人工干预
- **优点**：解耦、削峰、天然支持重试；符合团队技术栈（Kafka）
- **缺点**：需实现幂等控制与监控告警
- **适用性**：✅ 满足Q-2/Q-3/Q-8

#### 方案三：事件溯源（Event Sourcing）+ CQRS

- **描述**：状态完全由事件流重建，查询走独立读模型
- **优点**：极致可靠性与审计能力；天然支持回放与修复
- **缺点**：复杂度高，开发成本大，不符合CON-4（6个月交付）
- **适用性**：❌ 过度设计，不适合当前阶段

#### 方案四：双写数据库 + 消息队列（Write-Ahead Logging）

- **描述**：先持久化事件到DB，再异步发Kafka，后台拉取未发送事件补发
- **优点**：防止事件丢失，强一致性保障
- **缺点**：引入额外事务协调开销
- **适用性**：⭕ 可作为补充机制，但不宜作为唯一手段

#### 三、交互流程（关键路径）

1. 用户提交价格变更 → `PriceCommandService`

#### 步骤 4: 选择设计概念

**设计方案评估与选择**

### 方案一：传统虚拟机部署 + 手动发布 + 分散监控

- **描述**：使用VM部署各服务，Ansible脚本部署，Prometheus仅监控主机，日志本地查看
- **优点**：初期投入低，技术简单
- **缺点**：
  - 不满足Q-7（需改配置文件）
  - 无法实现快速回滚
  - 监控粒度粗，难以定位问题
  - 违背CON-6云原生倾向
- **适用性**：❌ 不符合战略方向

### 方案二：容器化部署 + Kubernetes编排 + GitOps + 统一可观测平台

- **描述**：
  - 所有服务打包为Docker镜像
  - 使用Kubernetes管理部署、扩缩容、滚动更新
  - Git作为唯一事实源，通过Argo CD实现GitOps自动同步
  - Prometheus + Grafana + Loki + Tempo构建统一观测栈
  - Jenkins/GitLab CI构建CI流水线
- **优点**：
  - 完全满足Q-7：镜像不可变，配置外置，跨环境一致性高
  - 满足Q-8：支持细粒度指标、日志、链路追踪
  - 支持CRN-5：自动化部署与回滚
  - 符合CON-2/CON-6：云原生标准实践
  - 提升可测试性（Q-9）：可在测试环境复现生产状态
- **缺点**：学习曲线较陡，初期搭建成本较高

### 方案三：Serverless架构（如AWS Lambda/Fargate）

- **描述**：将服务拆分为函数，事件驱动执行
- **优点**：极致弹性，按需付费
- **缺点**：
  - 对长连接、状态管理支持差（如价格计算可能超时）
  - 调试与监控复杂
  - 不利于团队现有Java/Kafka技术栈延续（CRN-2）
  - 初始迁移成本高
- **结论**：❌ 当前阶段不适用

### 最终选择：

✅ **方案二：容器化 + Kubernetes + GitOps + 统一可观测平台**

> **优势总结**：
>
> - 兼顾自动化、一致性、可观测性与团队技术延续性
> - 可逐步演进，支持未来向Service Mesh迁移
> - 成为行业标准，生态成熟，降低长期维护风险
